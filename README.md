# e2e-monitor

> 전반전부터 후반전까지, 당신의 터미널은 일하고 있었습니다.

월드컵이 개막했다. 그리고 당신에게는 오후 회의가 있다.
스코어 새로고침은 감흥이 없고, 영상 탭은 변명이 불가능하다. 그래서 만들었다 —
경기를 백그라운드에서 따라가다가, 머드 게임처럼 살아있는 텍스트 중계를
**당신의 직군 산출물로 위장한 형식**으로 로그 파일에 흘려보내는 Claude Code 스킬.

```
12:37:03 ✗ FAIL  RSA.goalkeeper.save() — 67' 히메네스, 박스 우측에서 한 번 접고
                 왼발 감아차기. 골키퍼 손끝을 스치고 골망이 출렁인다. 아스테카 폭발
12:37:03 ● deploy success → MEX 1 : 0 RSA (build #67, Estadio Banorte)
```

멀리서 보면 vitest 로그, 가까이서 읽으면 캐스터의 절규다.

## 설계 원칙

**3미터 / 30센티 원칙.** 3미터 거리에서 보면 업무 화면, 30센티에서 읽으면 머드 게임 중계.
형식(포맷, 타임스탬프, 용어)은 직군의 실제 산출물을 따르고, 내용(서사, 감정, 현장감)은
중계를 유지한다. 둘이 충돌하면 형식이 이긴다 — 위장이 우선이다.

**서사는 양념, 사실은 원본.** 스코어, 시간, 선수명은 원데이터 그대로. 각색은 묘사를
입히는 것이지 없는 어시스트를 지어내는 게 아니다.

**침묵도 위장이다.** 새 이벤트가 없으면 아무것도 출력하지 않는다.

## 동작 구조

```
┌─ daemon (백그라운드) ────────────────────────────────┐
│  ESPN unofficial API 폴링 (기본 10s)                  │
│    → 이벤트 tier 판정                                 │
│    → claude -p (headless)로 페르소나 스킨 각색         │
│    → ~/.e2e-monitor/match-{id}.log 에 append          │
│  tier-2 상황에선 폴링 3s 단축 + 라인 단위 즉시 flush    │
└──────────────────────────────────────────────────────┘

소비는 tail 하나로 끝:
$ tail -f ~/.e2e-monitor/match-760415.log
```

각색 주체는 daemon이다. Claude Code 세션이 떠 있을 필요가 없고, 세션은 본업에 쓰면 된다.
각색에는 `claude -p`(headless CLI)를 사용한다 — Claude Code 스킬이므로 CLI 존재를
전제하며, 경기당 호출은 주요상황 단위 30~50회 수준의 짧은 프롬프트다.
CLI가 없는 환경을 위한 템플릿 기반 폴백(머드 감성 열화 버전)을 제공한다.

## 중계 밀도 — 2-tier

| tier | 대상 | 동작 |
|------|------|------|
| 1 (기본 흐름) | 슛·위기/기회, 파울, 세트피스, 카드, 교체 | 10s 폴링, 상황당 1~2줄 |
| 2 (스트리밍 승격) | PK 선언, 골 전후 시퀀스, VAR, 80'+ 박빙 스코어 | 3s 폴링, 호흡 끊어 라인 단위 flush |

승격 판정은 ESPN commentary의 이벤트 type 기반 휴리스틱이며, 기준은
`config.json`에서 조정할 수 있다. (기본 임계값은 2026 조별리그 실경기 데이터로 확정했다.)

## 페르소나 스킨

`~/.e2e-monitor/config.json`의 `skin` 필드로 지정한다. 미지정 시 cwd를 보고 추정한다
(tsconfig.json이 보이면 `ts-dev`).

기본 제공: `ts-dev`(테스트 러너/CI 로그체), `planner`(요구사항 정의서체 — 골은 "승인 완료"),
`raw`(위장 없는 풀 머드).

스킨은 `skins/*.md` 마크다운 파일 하나로 추가된다. 형식 규칙과 예시 톤만 기술하면
각색 프롬프트에 주입되는 구조다. **당신의 직군 스킨을 PR로 보내달라** — 디자이너의
Figma 코멘트체, 데이터 분석가의 쿼리 로그체, 무엇이든.

## 골 애니메이션

위장에는 하나의 예외가 있다: **골**. 어차피 자리에서 환호할 순간이라면, 터미널도
환호해야 한다. 골이 터지면 약 10초간 위장을 해제하고 ASCII 연출이 로그를 타고
흐른다 — 공이 날아가고, 골망이 찢어지고, 대형 GOAL이 한 줄씩 차오른다.
끝나면 즉시 위장 포맷으로 복귀하고, 스코어·시간 fact 라인은 연출과 무관하게
항상 출력된다. 끄려면 config에 `"goalAnimation": false` (기본 on).

아트는 `art/goal/*.txt` 플러그인이다. 여러 팩이 있으면 골마다 랜덤으로 하나를
고른다. **당신의 골 세리머니를 PR로 보내달라** — 작성법은 `skins/README.md`.

## 사용법

```bash
# 오늘 경기 목록
npx tsx scripts/poll.ts list

# 데몬 시작 (경기 종료 시 자동 종료 + 최종 보고 후 사망)
npx tsx scripts/poll.ts daemon <eventId> &

# 중계 켜기 = tail
tail -f ~/.e2e-monitor/match-<eventId>.log
```

Claude Code 안에서는 스킬 트리거("모니터 돌려줘", "경기 어떻게 돼가")로
위 과정을 대신 시킬 수 있다.

## 데이터 소스와 한계

ESPN의 비공식(undocumented) API를 사용한다. 인증은 필요 없지만 공식 지원도, SLA도 없다.
언제든 스키마가 바뀌거나 막힐 수 있으며, 그 경우 daemon은 죽지 않고 raw JSON을
로그에 남기며 버티도록 설계되어 있다.

같은 이유로 폴링 간격 하한(10s/3s)은 코드에 고정되어 있고 설정으로 더 줄일 수 없다.
이 도구가 인기를 얻어 엔드포인트가 막히면 모두가 잃는다. 점잖게 쓰자.

마이너 매치는 분 단위 commentary 없이 keyEvents만 제공될 수 있다. 이 경우 중계 밀도는
낮아지지만 빅 이벤트 서사는 유지된다.

## Roadmap

네이버 스포츠 문자중계 폴백(한국어 원문 소스), 멀티 경기 동시 추적,
스킨 컨트리뷰션 가이드 문서화.

## Disclaimer

이 프로젝트는 ESPN과 무관하며, 비공식 엔드포인트의 가용성을 보장하지 않는다.
업무 시간의 사용으로 발생하는 모든 결과는 사용자의 책임이다. 우리는 도구를 만들었을 뿐,
당신의 스탠드업 발표 차례에 골이 터지는 것까지는 책임질 수 없다.

## [내부] v1 스펙 확정 사항 요약

대상: 공개 배포 (GitHub). 소비 패턴: 백그라운드 데몬 + tail.

각색 주체: daemon의 `claude -p` 호출 (세션 일괄 각색은 실시간성 요구와 충돌하여 기각,

페르소나: config 정본 + cwd 추정 보조, 스킨 플러그인(`skins/*.md`) 구조.

**완료 기준(v1)**: 데몬 시작 + tail만으로 한 경기 풀타임 동안 사람 개입 없이
2-tier 중계가 흐르고, 종료 시 데몬이 최종 보고를 남기고 스스로 종료.

## [내부] 구현 시 결정 기록 (2026-06-12, v1 구현하며 해소)

1. **tier-2 휴리스틱 임계값 — 확정.** 2026 조별리그 MEX-RSA(760415, 풀타임)
   raw 덤프(plays 1,313건 / commentary 102건, 44개 type)로 분류표 작성.
   - **tier-2**: 골 — type id 나열이 아니라 typeText `Goal*`/`Own Goal*` + 텍스트
     `Goal!` prefix로 판정 (ESPN이 골을 70 Goal / 137 Goal - Header 등 변형 type으로
     쪼개기 때문). PK — typeText/text `penalty` 매칭 (commentary의 penaltyKick 플래그는
     null이라 신뢰 불가). VAR — `VAR*` prefix (172 등 별도 id 패밀리). 레드카드 93.
     경기 경계 80/81/82/83 (킥오프/HT/후반시작/종료).
   - **tier-1**: 자연어 문장 텍스트 type — 66 Foul, 94 Yellow, 76 Sub, 95 Corner,
     106/117/135/136 슛 계열, 68 Offside, 122 Handball, 129/130 Delay.
   - **tier-0 (출력 스킵)**: "이름 (팀) 액션 at N'" 템플릿 노이즈 23개 type —
     118 Pass가 전체의 63%(828건). 미지의 type은 텍스트 패턴으로 동일 판정.
   - **80'+ 박빙**: summary header의 displayClock 분 ≥ 80 && |스코어차| ≤ 1 →
     상황 전체 tier-2 (추가시간 "90'+n'"도 자연 포함됨을 실측 확인).
   - 골/PK/VAR/레드 후 cooldownSec(120s) 동안 3s 폴링 유지 — "골 전후 시퀀스" 구현.
   - 구현: `src/tier.ts`. config은 type 승격만 허용, 강등으로 골을 숨기는 건 미지원.

2. **`claude -p` 지연 — 실측 결과 동기 각색 폐기.** 단문+haiku 6.8s, 중앙값 10.95s로
   3s 주기 초과 확정 (CLI 프로세스 기동 고정비가 지배적이라 프롬프트 단축으로 해결 불가).
   "템플릿 우선 출력 + 사후 각색 교체"로 전환하되, append-only 로그를 tail로 소비하는
   구조에서 이미 출력된 라인의 **교체는 불가능** → "사후 각색은 보강 라인으로 append"로
   재해석해 채택:
   - **tier-2**: 탐지 즉시 템플릿으로 flavor+fact 출력(지연 0). 각색은 비동기 단건
     호출로 진행, 완료 시 `replay.flavor` 라인(ts-dev는 `· retrace`)으로 사후 합류.
     동시 각색 2건 제한, 몰리면 조용히 포기 (침묵도 위장).
   - **tier-1**: tick에 쌓인 이벤트를 **배치 1회 호출**로 각색해 기동 고정비 상각,
     timeout(기본 25s)·실패 시 스킨의 카테고리별 폴백 desc 사용. fast mode 중엔
     케이던스 보호를 위해 tier-1도 템플릿 직행.
   - 사실 불변의 구조적 보장: fact 라인은 항상 데이터에서 직접 렌더되고 claude를
     통과하지 않는다. 각색은 flavor의 `{desc}` 한 칸에만 들어간다 (`src/render.ts`).
   - 부차 실측: summary 응답 cache-control max-age=7 → 3s 폴링의 실효 갱신은 ~7s.
     하한 3s는 스펙대로 유지 (서버 캐시가 한 번 더 막아주는 셈이라 부담도 없음).

3. **README 톤 — 유머 버전 유지.** 순화 버전 전환은 사용자 결정 사항이라 보류 그대로.
   본문 변경은 사실 갱신 2곳뿐 (임계값 "확정 예정" → 확정, Roadmap에서 해당 항목 제거).

4. **v0 poll.ts — 부재 확인, 신규 구현.** 워크스페이스·세션 기록 어디에도 v0 코드가
   존재하지 않음을 확인 (README 언급만 존재). diff/상태 관리는 `src/state.ts`로 신규
   구현: commentary `sequence` 기반 seen 셋 + 디스크 영속화(재시작 시 재방송 방지),
   keyEvents-only 매치는 `ke:{id}` 키. watch 커맨드는 만들지 않고 daemon 모드만 제공.

5. **(추가 결정) 폴링은 summary 1콜/tick.** commentary 항목에 play.type이 실려 있어
   (760414/760415 실측) 상태+스코어+commentary+keyEvents가 한 응답에 모두 포함 —
   core plays 피드·scoreboard 병행 불필요. 장애 시 raw JSON은 위장 로그를 오염시키지
   않도록 사이드카 `match-{id}.raw.jsonl`에 기록하고 exponential backoff(상한 60s)로
   버틴다. 데몬을 경기 중반에 시작하면 전반전 전체 재방송 대신 캐치업 요약
   (킥오프 + 골/퇴장/PK + HT 라인)만 출력한다.

    골이 아닌 정정 이벤트로 (`daemon.ts`, `state.ts`)
  - 합성 골이 무조건 홈팀 득점으로 렌더 → 스코어 델타로 팀 귀속, 판별 불가 시
    팀 표기 없는 generic 강등 — 틀린 사실을 찍지 않는다 (`render.ts`)
  - claude CLI가 SIGTERM 무시·파이프 점유 시 데몬 영구 동결 → SIGKILL + watchdog
    (`narrate.ts`). tick 본문 전체 try/catch, append 실패 무시 — "죽지 않는다" 보장
  - 스킨 파서 무음 실패 계열(단일 ini 블록만 파싱, guide가 섹션 순서 의존,
    wrapIndent 미검증 → 무한 루프/크래시) → 멀티 블록 병합, lookahead 완화,
    0~60 clamp, stderr 경고. 스킨 작성 스펙은 `skins/README.md`로 문서화
  - config 사용자 입력 전반 모양 검증(typeIds 배열 강제 + String 매핑 등),
    같은 경기 데몬 중복 실행 방지 lockfile, commentary sequence 기준 정렬
    (clock 없는 FT 항목이 골보다 먼저 나가는 순서 역전 방지), 최종 보고
    하이라이트 state 영속화(재시작 생존), pre-match 라인업 commentary 침묵 처리
- **잔여 리스크(수용)**: summary cache-control max-age=7로 3s 폴링의 실효 갱신 ~7s.
  골 텍스트 팀명 매칭 실패 시 이중 중계 억제가 못 잡는 엣지 존재(스코어 비교가
  보조 방어). keyEvents↔commentary 소스 플립 시 한 tick 분량 이벤트 흡수(유실) —
  골은 스코어 변화 합성이 보강.

## [내부] 골 애니메이션 결정 기록 (2026-06-12)

- **위장 해제 예외는 골 한정 — 사용자 결정 확정 사항.** 레드카드·PK 선언 등 다른
  tier-2는 기존 위장 그대로. config `goalAnimation`(기본 on)으로 끌 수 있고,
  끄면 기존 출력과 완전 동일.
- **아트 스타일: 점진 공개 채택, 프레임 반복 기각.** append-only 로그 + tail -f
  제약에서 '움직임'은 새 줄이 나타나는 리듬으로만 만들 수 있다(커서 제어·화면
  지우기 불가). 후보 비교:
  - *프레임 반복*(같은 그림을 위치만 바꿔 재출력): 터미널에선 이전 프레임이
    지워지지 않고 그대로 쌓인다 — 잔상이 연출이 아니라 스크롤 쓰레기가 된다. 기각.
  - *대형 아트 일괄 출력*: 한 번에 다 나오면 정지화면이라 '10초의 사건'이 안 된다. 기각.
  - *점진 공개*(채택): 공 궤적 가속 빌드업 → 임팩트 → 대형 GOAL 블록을 행 단위로
    차오르게 → 에코 감쇠 → 위장 복귀. 줄 단위 등장 자체가 모션이고, 끝난 뒤
    화면에 누적된 결과물이 한 장의 완성된 포스터로 남는다 — tail을 놓친 사람도
    스크롤백으로 정점을 본다.
- **색: yellow 단색 + dim/cyan 보조.** tail은 ANSI를 통과시키므로 사용. 다색
  무지개는 10초짜리 사건의 긴장감을 죽여서 단색 강조로. 색 쓴 줄은 파서가 reset을
  강제로 닫는다 — 애니메이션이 이후 위장 라인을 물들이면 위장 실패다.
- **케이던스 보호**: 데몬에선 tick 정규 출력이 모두 끝난 뒤 fire-and-forget으로
  시작 — 폴링·탐지·출력을 1ms도 막지 않는다. 대가로 다음 tick 라인이 프레임
  사이에 끼어들 수 있는데, 케이던스 > 연출 순수성으로 수용. replay는 순차 재생이라
  await로 막고(다음 이벤트는 어차피 페이스 대기), retrace 각색도 애니메이션 뒤에
  시작시켜 프레임 오염을 없앴다 — 데몬엔 허락되지 않는 사치.
- **내성**: celebrateGoal은 어떤 경로로도 throw하지 않는다(전체 try/catch + 프레임
  append 개별 무시). 아트팩 파서는 깨진 파일 단위로 격리, 전부 깨지면 내장 폴백.
  프레임 40개·딜레이 50~2000ms clamp·총 15초 비례 축소로 폭주 팩도 무해화.
  10초 내 연속골은 두 번째 연출을 조용히 포기(inFlight 가드) — 프레임 교차 출력은
  난장판이라 침묵이 낫다. fact 라인은 애니메이션 시작 전에 이미 나가 있다(순서:
  flavor → fact → 애니메이션) — 사실 불변 원칙은 구조적으로 유지.
- **캐치업 골은 애니메이션 없음**: 중반 합류 요약에서 과거 골 3개에 연출 3방을
  몰아치면 위장도 페이스도 깨진다.
